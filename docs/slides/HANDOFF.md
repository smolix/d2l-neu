## 1. Decisions already made (do not relitigate)

- **Diagram engine = plain JS, run under Node.** The engine (now at the repo
  root `diagrams/`) is pure string-building — *no DOM, no browser*.
  `node diagrams/render.mjs --out img/auto` writes SVG files directly.
- **Diagrams live in `img/auto/<id>.svg`** and are **committed**. They are
  intended to also upgrade the **HTML book and PDF** figures later, so they must
  be real rendered files (not runtime-only). `@fig:<id>` inlines them into a
  deck via a pandoc `{=html}` raw block.
- **Design diagrams portrait, and match the cell.** Wide-short figures are tiny
  in a column — stack panels vertically (see `concat`, `saving-memory`). Draw
  the actual values the executed cell shows. Geometric figures (arrows, angles,
  lengths) are encouraged.
- **Slides-only for now.** Do **not** touch the chapter body prose, the book
  figures, or `_quarto.yml` page layout in this phase. (A separate later pass on
  another server handles book/PDF integration and page-layout fixes.)
- **One section at a time, with a human review gate.** Generation is expensive
  and content needs human eyes. Propose an **outline first**, get sign-off, then
  produce the full block. Expect to **replace** the existing `<!-- slides -->`
  block for that file.
- **Stale content is a stop, not a workaround.** Some chapters are seriously out
  of date. If the executed notebook errors, uses deprecated APIs, or teaches
  something obsolete, **flag it for human rewrite** and stop — do not paper over
  it in slides. (Author-driven content rewrites are out of scope for you.)
- **Page-layout polish is deferred.** Note overflow/fit problems you see, but
  don't chase them this phase.

## 2. How the repo's slide pipeline works (use it; don't reinvent)

Source of truth is the chapter **`.md`** file. **Never edit `.qmd`** — they are
generated by `tools/d2l_preprocess.py` and overwritten.

**Where slides live.** At the end of each `.md`, after the content, in a block:

```markdown
<!-- slides -->

::: {.slide title="Reshaping"}
`reshape` rearranges the same elements into a different shape.

@ndarray-getting-started-5

A 12-element vector becomes a $3\times4$ matrix — no data is copied.
:::
```

**The grammar inside a slide div:**

- Plain Markdown prose → rendered verbatim (this is the *slide caption*, which
  you write — concise, not copied from the chapter body).
- `@<id>` → includes source code cell `<id>` **and** its executed output.
- `@!<id>` → output only, no code echo.
- `@<id>@<fw>` / `@!<id>@<fw>` → force a framework-specific variant.
- `. . .` → a reveal **fragment** break (incremental reveal).
- `::: {.fragment}` and other nested divs → passed through verbatim.
- Heading attrs: `title="…"`, `layout="2col"` (→ CSS class `.slide-2col`; also
  `.slide-figure`, `.slide-code`), `transition="…"`, `data-background*`.

**Cell ids.** Every Python fence in the source carries a stable `#<id>` assigned
by `tools/add_cell_ids.py` (idempotent; ids **never change**, even on rename or
move). Untagged Python cells apply to all frameworks; `#@tab pytorch` etc. scope
a cell; `#@tab all`/`%%tab all` are stripped from the corpus.

**Build / preview (slides are CPU-only, no GPU):**

```bash
# Render one file's deck for one framework while iterating:
make -B slides-pytorch SLIDES_FILTER=chapter_preliminaries/ndarray.md
# All frameworks, parallel-safe:
make -j4 slides
# Live preview while editing the slide block:
tools/watch_slides.py --fw pytorch --file chapter_preliminaries/ndarray.md
```

Outputs are injected by `tools/inject_outputs.py slides`, matched **by cell id**
against `_notebooks/<fw>/…`; slide decks render with `eval: false`. Decks are
emitted by `tools/gen_slides.py` (`::: {.slide}` → Quarto `revealjs` qmd →
`quarto render --to revealjs`). `tools/audit_slides.py` checks slide↔cell
integrity.

**Styling** is entirely in `_d2l-slides.scss` (palette tokens, In/Out cell
output, blockquote callouts, `2col`/`figure`/`code` layouts, the `.d2l-deck-nav`
top bar). **Put visual changes there, never in a per-deck stylesheet.**

## 3. Conceptual style (what makes a deck good)

The reference deck and §2.1 block embody these rules. Apply them:

1. **One idea per slide.** No walls of text, no walls of code.
2. **Follow the notebook's teaching sequence**, but **curate**: skip cells that
   don't teach; you are not re-telling the whole chapter.
3. **Trim noisy output.** Prefer the clean illustrative example over a literal
   dump (e.g. show `randn(3,4)` not a 24-number `zeros((2,3,4))`; show `exp` on a
   small vector, not an overflow). Outputs you *do* show must match the executed
   notebook faithfully.
4. **Write your own concise captions.** Never paste chapter-body paragraphs into
   a slide. One or two lines that frame the cell.
5. **Stage with fragments (`. . .`)** when a slide builds in steps (e.g. show
   code, then its consequence).
6. **Give every structural/spatial idea a diagram** (shapes, indexing regions,
   broadcasting, memory aliasing, data flow). Text can't carry these well.
7. **Bookend the deck:** a "basics/why" opener and a "recap" closer (the §2.1
   block calls them *Tensor Basics* and *Recap*).
8. **Callouts** (blockquotes) for rules and warnings — sparingly.
9. **Audience: strong undergrad.** Assume programming maturity; explain the ML-
   specific intuition.

## 4. Visual style (matches `_d2l-slides.scss`, already in repo)

- **Type:** Source Sans 3 (body/headings), JetBrains Mono (code). 30px root.
- **Palette:** blue `#2196F3` (primary/nav), link `#1976D2`, code/identifier
  accent purple `#7D12BA`, ink `#15181C`/`#3A4049`, white bg. (Full token list
  at the top of `_d2l-slides.scss` and mirrored in `diagrams/engine.mjs` `C`.)
- **h2** carries the blue underline (section-divider feel). Code blocks have the
  light blue left rail. Outputs render in the muted `.cell-output` style.
- **Layouts:** `layout="2col"` to set a diagram beside its code; `figure` for a
  full-bleed diagram; `code` for code-dense slides. NB: `2col` is a CSS grid that
  flows the section's children across two columns in order — getting a clean
  *code | diagram* split can need a layout tweak; that polish is deferred.
- **Polish deltas from the deck not yet in scss** (the In/Out *card* wrapper,
  cover/divider slide treatments, `callout.rule`/`.warn` variants, kicker
  labels): if you want them, **add them to `_d2l-slides.scss` as classes** — do
  not inline CSS in a deck.

## 5. Diagrams — the main new work

### 5.1 Author

Diagrams are JS functions returning an SVG string, built from the engine
helpers (`grid`, `arrow`, `block`, `chip`, `tx`, `rc`, `svg`) in
`diagrams/engine.mjs`. See `diagrams/ndarray.mjs` for eight worked examples and
`diagrams/README.md` for the helper reference and a "how to add one" recipe.

- **One function per diagram**, keyed by a **stable id** using
  `<chapter-file>-<concept>` (e.g. `ndarray-broadcasting`). The id is the SVG
  filename and the slide reference. **Never rename an id** once used.
- Add a new chapter's module to `diagrams/registry.mjs`.
- Use the engine's `C` color tokens — never hard-code hex outside the engine —
  so a palette change restyles everything. (`C` mirrors the scss tokens; keep
  them in sync, or generate `C` from the scss later.)

### 5.2 Render

```bash
node diagrams/render.mjs --out <repo>/img/auto            # all
node diagrams/render.mjs --out <repo>/img/auto ndarray-broadcasting   # one
node diagrams/render.mjs --list                           # known ids
```

Writes standalone `img/auto/<id>.svg` (XML namespace + width/height; ready for
`<img>`, inline, and rsvg→PDF). **Commit the SVGs** (reviewable diffs, no reader
build dependency).

### 5.3 Reference from a slide

Recommended (exact fonts, clean source): a small `gen_slides.py` enhancement —
a `@fig:<id>` placeholder that **inlines** `img/auto/<id>.svg` into the deck so
it inherits the page's loaded fonts. ~10 lines next to the existing `@<id>`
handling in `_resolve_body`.

Works **today with no tooling change** (slight font fallback, since an `<img>`
doesn't inherit page fonts): a plain Markdown image line inside the slide div —

```markdown
::: {.slide title="Broadcasting" layout="2col"}
When shapes differ, size-1 axes **stretch** — no data copied.

@ndarray-broadcasting-1

![](img/auto/ndarray-broadcasting.svg)
:::
```

### 5.4 Fonts in static SVG (fidelity caveat)

Each label's `font-family` ends in a generic family, so a static SVG always
*reads* correctly. For pixel-exact Source Sans 3 / JetBrains Mono in `<img>`-
embedded or PDF contexts you'll later want to either inline the SVG (slides) or
embed `@font-face` (book/PDF). Flagged for the later book/PDF pass — don't solve
it now.

## 6. Framework variants (PyTorch / JAX / TF / MXNet)

The decks are generated **per framework** from one shared slide block; code and
output swap by cell id. Most differences are already handled by the existing
`#@tab`/`@<id>@<fw>` mechanism. Watch for cases where the *framing* differs:

Scope a divergent slide with `only="fw[,fw]"` / `except="fw[,fw]"` (now
implemented) and give the diagram an fw variant. Worked example in
`ndarray.md`:

- **JAX immutability.** No in-place writes; `x.at[idx].set(...)` returns a new
  array, and `saving-memory-2` has *no output* (a comment-only cell). The
  *Writing* and *Saving-memory* slides are `only="jax"` with their own framing
  and the `ndarray-saving-memory-jax` diagram.
- **TensorFlow** uses `tf.Variable` + `tf.function` for the in-place story, and
  `Tensor`s are immutable — `only="tensorflow"` slides.
- **NumPy round-trip** shares memory in PyTorch but **copies** in MXNet/TF/JAX —
  scoped, since the "one shared buffer" framing (and diagram) is PyTorch-only.
- **Randomness** (JAX PRNG key), **eager vs graph**, **dtype defaults** — keep
  captions neutral where possible; scope only where the *concept* changes.

Decks that are pure code/output swaps need **no** scoping (e.g.
`linear-algebra` used zero `only=`/`except=` slides). Always check by reading
all four `outputs/<fw>/<chapter>/<file>.json` before deciding.

## 7. Per-section workflow (the loop)

1. **Read** the section's `.md` (source, incl. its current `<!-- slides -->`
   block if any) **and** the executed `_notebooks/<fw>/<chapter>/<file>.ipynb`
   for each framework. Note the cell ids and their outputs.
2. **Triage staleness.** If the notebook errors / uses dead APIs / teaches
   something obsolete → write up what's wrong, flag for human rewrite, **stop**.
3. **Draft an outline** — slide titles, which cell id each shows, where diagrams
   go, which slides are framework-specific. **Get human sign-off on the outline**
   before writing the full block (cheap checkpoint; saves tokens).
4. **Author the `<!-- slides -->` block**: titles, concise captions, `@id`
   refs in notebook order, `. . .` fragments, `layout=` where a diagram pairs
   with code, framework scoping.
5. **Author + render any new diagrams** (`diagrams/<file>.mjs` →
   `node render.mjs --out <repo>/img/auto`), reference them in the slides.
6. **Build & check:** `make -B slides-pytorch SLIDES_FILTER=<file>.md`, then the
   other frameworks; run `tools/audit_slides.py`.
7. **Self-review** against §8, and eyeball against `north-star.html`.
8. **Hand to human**, iterate, then replace the old block.

## 8. Acceptance checklist (run before handing off)

- [ ] No wall of code or text on any slide; one idea per slide.
- [ ] Code cells appear in notebook order; non-teaching cells dropped.
- [ ] Shown outputs match the executed notebook; noisy outputs trimmed.
- [ ] Captions are original and concise — no pasted chapter prose.
- [ ] Every structural/geometric concept has a diagram; diagrams use engine
      tokens and **match the cell's values**.
- [ ] Fragments (`. . .`) used where a slide builds in steps — and **never
      inside a `::: {.col}`** (they render as a literal "...").
- [ ] Cover, section dividers, kickers, In/Out cards present.
- [ ] Per-framework *framing* checked across all four `outputs/<fw>/…json`;
      divergent sections use `only=`/`except=` (esp. JAX immutability, TF
      `Variable`/`tf.function`, NumPy shared-vs-copy).
- [ ] **No slide overflows 720 px** — run the overflow sweep
      (`docs/slides.md` → *Quality rules*); fix by trimming / `@-` code-only /
      splitting / widening, not per-slide scrollbars.
- [ ] Renders cleanly in **all four** frameworks; `audit_slides.py` clean.
- [ ] Visually consistent with `north-star.html`.

## 9. Updating a deck after a notebook re-run

Code/output already re-propagate: `inject_outputs.py slides` re-matches by cell
id, and Make stamps (`.executed` → slides) drive incremental rebuilds
(`SLIDES_FILTER` narrows to one file). So a code edit + re-run + `make slides`
refreshes the deck with **no slide-source change**.

Two things to mind:
- **Value-bound diagrams** (ones that draw real numbers — e.g. a computed
  result or sampled values) must be regenerated too; have them read values from
  `_notebooks/<fw>/…` by the same cell id. Most diagrams are *structural*
  (shape-only) and never change.
- **Drift:** if a cell id vanished or its output shape/type changed, the slide's
  framing may now be wrong. Extending `audit_slides.py` to warn on this is the
  recommended early-warning (full auto-fix is out of scope).

## 10. Gotchas

- Source `.md` is truth; **never edit `.qmd`**.
- `make lib` (rebuilding `d2l/*.py` from `#@save`) can affect notebooks beyond
  the edited file/framework — rerun all plausibly affected notebooks.
- The diagram engine is **DOM-free**: `node render.mjs` needs no browser. Don't
  add Playwright for diagrams.
- Keep `diagrams/engine.mjs` `C` in sync with the scss palette tokens.
- **Reserved class names:** don't author `::: {.callout}` (Quarto turns it into
  its own callout component) or `.columns`/`.column` (Quarto's grid). Use
  `.d2l-note` and `.cols`/`.col`.
- **`. . .` is top-level only** — inside a `::: {.col}` it renders as text.
- **Inline SVGs need a `{=html}` raw block** or markdown parses `[…]`/quotes
  inside them; `@fig:` already does this.
- **Subscripts:** unicode `ⱼ` is missing in many fonts — use SVG
  `<tspan baseline-shift="sub">` for true subscripts.
- **Don't add `scrollable: true`** to the deck config, and don't pin a
  `height`/`min-height` on `.slides`/sections — both fight reveal's
  fit-to-window scaling. The navbar is scaled to match in the overlay.
- **Self-review screenshots** (Playwright): set
  `Reveal.configure({transition:'none', fragments:false})` and navigate with
  `Reveal.slide(0, v)` — our decks are one vertical stack (the deck H1 wraps the
  `##` slides), so the horizontal index is always 0. Use a `?v=N` cache-buster;
  a missing `favicon.ico` 404 is harmless.
- Generation is token-expensive → the outline checkpoint in §7.3 is not
  optional.
