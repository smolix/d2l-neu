# Review — chapter_multilayer-perceptrons/mlp-implementation.md  (§5.2 "Implementation of Multilayer Perceptrons")

**Role in the chapter:** Bridges theory (§5.1 mlp.md) to practice: builds a one-hidden-layer MLP on Fashion-MNIST twice — once from scratch to expose parameter management and the forward computation, once with `nn.Sequential` to show the framework payoff. It is deliberately narrow: no new theory, no training-loop mechanics (those are inherited from §3–4).

**Verdict:** The file does its narrow job cleanly and is free of errors. The from-scratch section correctly isolates what is *new* vs. softmax-scratch (two sets of params, the ReLU, the composition). The concise section is idiomatic for its era. The main deficits are (1) a stale, deflating Summary that undersells the milestone and misdirects the reader toward §5.3–5.5 without naming them; (2) `torch.clamp(X, min=0)` as a ReLU — functionally correct but pedagogically backward (`torch.relu` is idiomatic and the name communicates intent); (3) the prose explaining why `pow-of-2` widths matter is slightly imprecise; (4) the sigma=0.01 initialization is used without a single sentence deferring to §5.4 — a reader naturally wonders "why 0.01?"; (5) MXNet should be tombstoned. No math errors, no broken rendering, training converges to ~0.8 val_acc across all four frameworks.

**Grade:** B+. Assignable, but the Summary is weak enough to leave students with no motivating bridge to the rest of the chapter, and the MXNet-as-live-peer framing is misleading in 2026.

## 2. Teaching quality

### Structure & flow

The file has two `##` sections (scratch, concise) plus Summary and Exercises — appropriate for a short implementation section. The three-level subsections (`### Initializing Model Parameters`, `### Model`, `### Training`) mirror the softmax scratch file, which is intentional and good for pattern-matching. No restructuring needed.

The one flow issue: the "Training" subsection under "Concise Implementation" is a single code cell and three sentences. This is correct — the point *is* that training is identical — but the prose says "The training loop is exactly the same as when we implemented softmax regression." It then adds a modularity observation. This is fine; no change needed here.

### Figures

**No illustrative figures are present.** The only figures are the four training-curve SVGs produced by `d2l.plot`. For an implementation section this is acceptable — the training curves are computed results that legitimately belong in cells. However, a simple architecture diagram (784 → 256 → ReLU → 10) would help readers who haven't yet internalized what a two-layer MLP looks like. This is a *nice-to-have*, not a blocker.

The slides section (out of scope for this review, per the guide) already has an ASCII diagram:
```
  X (batch, 784)
       │ Linear  784 → 256
       │ ReLU
       │ Linear  256 → 10
       ▼
  logits (batch, 10)
```
A clean SVG version of this (generated by the mdl-figure infrastructure) would be a worthwhile addition as a pre-generated figure. See `MLPI-6`.

### Prose & clarity

**Line 59 — "powers of 2" explanation is imprecise.**
Current text: *"Typically, we choose the layer widths to be divisible by larger powers of 2. This is computationally efficient due to the way memory is allocated and addressed in hardware."*
Problem: The efficiency comes primarily from **SIMD/tensor-core alignment** in matrix multiplication kernels, not from general memory allocation. Misframing this as "memory allocation" plants a misconception. Fix: change to reference kernel alignment (see MLPI-5).

**Lines 181–182 — run-on prose:** "Finally, we implement our model with just a few lines of code. Since we use the framework built-in autograd this is all that it takes." Two issues: (1) *built-in* should be *built-in* (hyphenated, or reworded); (2) the sentence is saying that autograd is doing the work — this is worth making explicit rather than implied. A short improvement: "Since autograd tracks the computation graph through these primitives, we need write only the forward pass."

**Lines 298–303 — Summary is weak:** The current Summary rambles about difficulties of from-scratch naming and then says "you have now reached the state of the art of the late 1980s." This is *deflating* — students just learned something real and useful, and the Summary dismisses it as 1980s technology. More importantly, it does not name what comes next. See MLPI-1.

### Exercises

The exercises are serviceable but not excellent.

- Exercises 1–3: Good — vary width, add a layer, analyze single-neuron layer. Exercise 3 ("Why is it a bad idea to insert a hidden layer with a single neuron?") is a genuinely thought-provoking question; it has more than one answer (bottleneck, gradient flow, expressiveness). Keep all three.
- Exercise 4: Learning rate sweep is standard; fine.
- Exercise 5: Joint hyperparameter optimization is good but part (c) asks students to "describe an efficient strategy" — this deserves a hint that Bayesian optimization / random search are the standard answers, otherwise it is too open-ended for a textbook exercise. A small edit: add "(*Hint:* what is the advantage of random search over grid search?)" 
- Exercises 6–7: Speed benchmarks and memory-alignment experiments are interesting but feel disconnected from the MLP narrative. Exercise 7 in particular (1024/1025/1026/1028/1032 matrix sizes, measuring bus width) is a hardware-engineering exercise, not an MLP exercise. Worth moving to Builders' Guide or cutting.
- Exercise 8: "Try out different activation functions" — should at minimum mention ReLU, sigmoid, tanh, and now GELU, pointing at `torch.nn.functional.gelu` or `jax.nn.gelu`.
- Exercise 9: "Is there a difference between weight initializations?" — this is the only hint that init matters, but it is too vague. After MLPI-2 adds the init deferral sentence, this exercise should be updated to say "Compare `sigma=0.01`, `sigma=0.1`, and Xavier/Glorot initialization; how do they affect convergence?" This makes the exercise concrete and bridges to §5.4.

## 3. Code & examples

### Does the code teach?

Yes. Every cell earns its place:
- The init cell teaches that two-layer MLP = two W + two b, and shows how each framework registers parameters.
- The `relu` cell is a 1-line illustration that activates the name-to-operation connection.
- The `forward` cell shows the exact computation: flatten → affine → relu → affine. The equations in §5.1 are now code.
- The concise cell shows the framework abstraction pays off in brevity.
- Both training cells show that the training loop is framework-agnostic.

No cells draw schematic figures; all cells compute. Correct.

### PyTorch

- **`torch.clamp(X, min=0)` is non-idiomatic.** The standard idiom is `torch.relu(X)` (or equivalently `F.relu(X)`). Using `clamp` is functionally correct but pedagogically backward: the cell's stated purpose is to show "how ReLU works," and `clamp` hides that name. The comment says "we implement the ReLU activation ourselves," which makes sense, but even self-implementing ReLU, `torch.maximum(X, torch.zeros_like(X))` or just `X * (X > 0)` would be more transparent. The cleanest option: keep the manual cell as `torch.maximum(X, torch.zeros_like(X))` (matches the MXNet/TF/JAX idiom exactly: all three use `maximum(X, 0)`). See MLPI-4.
- **`nn.LazyLinear`**: Still correct and idiomatic in PyTorch 2.x. No change needed.
- **`nn.Sequential(nn.Flatten(), nn.LazyLinear(num_hiddens), nn.ReLU(), nn.LazyLinear(num_outputs))`**: Clean and idiomatic. Good.
- Outputs: Training curves converge (~0.8 val_acc at epoch 10 for both scratch and concise). No warnings in the manifests.

### JAX

- **`from flax import linen as nn`**: Flax Linen is not deprecated in the sense that it is removed — it still works as of Flax 0.9+. The book's `d2l/jax.py` itself imports `flax.linen`, so changing this in chapter files alone would be inconsistent. This is a book-wide decision (whether to migrate to Flax NNX), not a per-file fix. Flag for the overview but do not change in this file.
- **`self.param('W1', nn.initializers.normal(self.sigma), ...)`** in the scratch class: correct Linen API. The `setup()` pattern is preferred over `@nn.compact` for explicit parameter naming (as is done here). Good.
- **`@nn.compact` in the concise class**: Using `@nn.compact` for the `__call__` method is also idiomatic Linen. Layers created inline (`nn.Dense(...)`) are named by order. No issue.
- **`nn.relu(X)` in the concise class**: `nn.relu` in Linen is `flax.linen.activation.relu`, which is `jax.nn.relu` under the hood. Correct, though `jax.nn.relu(X)` would be slightly more transparent.
- Outputs: Converge similarly to PyTorch. Clean.

### TensorFlow

- **`tf.keras.models.Sequential`**: In Keras 3 (2023+), the preferred import is `keras.Sequential` not `tf.keras.models.Sequential`. Since the book currently uses `tf.keras` throughout (check `d2l/tensorflow.py`), this is again a book-wide migration question, not a file-level fix. Flag for overview.
- **`tf.keras.layers.Dense(num_hiddens, activation='relu')`**: Correct. Clean.
- **`tf.keras.layers.Flatten()`**: Correct.
- Outputs: Clean convergence.

### MXNet

**MXNet was retired by the Apache Software Foundation on September 20, 2023; repository archived November 17, 2023; transferred to Apache Attic February 2024.** Presenting MXNet as one of four equal framework choices in 2026 is actively misleading to students. The book-wide action is to tombstone the MXNet tabs (see MLPI-3). In this file specifically:
- The imports cell has a MXNet variant.
- The `MLPScratch` init uses `np.random.randn` (MXNet's numpy) and manual `param.attach_grad()`.
- The concise `MLP` uses `nn.Dense(activation='relu')` and `self.net.initialize()`.
None of these are issues to fix in isolation — they should be removed or marked legacy when the book-wide MXNet tombstone is applied.

### Cross-framework consistency & d2l conventions

- **PyTorch relu divergence**: All other frameworks use `maximum(X, 0)`. PyTorch uses `torch.clamp(X, min=0)`. This is unnecessary divergence for a cell whose purpose is to show equivalence. Fixing to `torch.maximum(X, torch.zeros_like(X))` (matching the other three) would be cleaner. See MLPI-4.
- **Imports cell**: One import block per framework near the top. Correct; no re-imports later.
- **Cell IDs**: Stable across all frameworks. Good.
- **`d2l.reshape`, `d2l.matmul`**: Used in the all-framework `forward` cell. These resolve to the correct framework-specific ops. Correct.
- **Training cell (`mlp-implementation-training`) is all-framework (no `%%tab`)**: The training call is `trainer.fit(model, data)` — genuinely framework-agnostic code. The `model` and `data` objects are framework-specific (set in each tab above), so this is fine.
- **`trainer` reuse across scratch and concise**: The concise training cell reuses the `trainer` object from the scratch section without reinitializing it. This is intentional (to show modularity) but could confuse a student who runs cells out of order. Low-priority note.

## 5. Keep — what is already excellent (do not lose this)

- **From-scratch section isolates exactly the right delta.** The `MLPScratch` class adds precisely what is new vs. `SoftmaxRegressionScratch`: two W/b pairs and a ReLU. It does not re-explain the training loop, the loss, or data loading. This discipline is excellent and should be preserved.

- **`forward` cell is elegantly minimal.** Three lines (`reshape`, `H = relu(...)`, `return matmul(...)`). The computation graph is completely legible. This is the pedagogical high point of the file.

- **Concise section correctly explains what `Sequential` buys and why there is no `forward` override.** The explanation at lines 267–282 — that `Module.forward` delegates to `self.net(X)` — is honest and useful. Many textbooks skip this and leave students wondering why the concise class works without a `forward` method.

- **JAX scratch section uses `setup()` correctly.** Using `setup()` with explicit `self.param(...)` calls (rather than `@nn.compact`) for the from-scratch class is the right pedagogical choice: it makes parameter ownership visible.

- **Training convergence is real and clean across all four frameworks.** The ~0.8 val_acc result confirms the model trains correctly and provides a concrete performance baseline. The plots (train_loss, val_loss, val_acc) are exactly the right metrics to show.

- **The "modularity" observation in the concise Training section is correct and worth keeping.** "This modularity enables us to separate matters concerning the model architecture from orthogonal considerations." This is the core software-engineering argument for the framework abstraction and earns its place.
